IGB200

Overview

A game was to be made in a group of 4 that followed the theme of QUT Open day.

Our group had chose to do an island based survival game where the character would build skills relating to QUT in order to escape the island and make QUT O'Week in time.

The execution of this was poorly done and to be honest did not reflect the initial scope of the project well, however the game itself was functional and showed unique gameplay over others, most notably being a tile-based game.

What was learnt

Development of the project

In early development, a unity preset was loaded and a character was added. The only real functionality was the ability to walk and collect sticks from trees.

Eventually textures were added along with the ability to collect multiple items. A title screen was finally created and cleaner UI was added after.

A campfire along with several assets were created by me to make the environment more engaging.

My favourite contribution is the dynamic tiles where each water tile would connect and be animated along with the tiling nature of everything else.

Possible Improvements

The character asset was made by another member and felt out of place from the rest of the assets.

One of the biggest issues was I was responsible for almost all asset creation and a lot of the core gameplay mechanics. If I had the option to chose one I would have stuck with asset creation to make the game look and feel as smooth as possible.

A game like this with tweaks made to remove the initial theme of O'Week has high potential, and could be worked on or even rewritten in future.

IGB383

Overview

IGB383 consisted of 3 incomplete projects and each project had to be improved using navigation algorithms.

The first project was a pacman-like game with nodes for the enemies to follow.

The second was a space fighting game where both the missiles and the drones had to do a series of movements depending on various values.

A competitive bot was the last assignment, however I did not engage in this project.

What was learnt

Path navigation using A* pathfinding and what is denoted as a greedy search, where the largest node is taken without any other consideration.

A comparison between the two was written along with finite state diagrams.

The second project covered a "Bee's Algorithm" which is an algorithm to simulate flying agents as if they were a colony.

Other niche topics such as Boid behaviour, models on heuristic integer values and such were also learnt.

Development of the project

The first project was quite simple, first boundaries for the enemies were added where enemies would either attack or hide from the player.

then each node was connected, which would serve as the navigation for each enemy. values are weighted based on either the A* or greedy algorithm to get to the player.

The second project involved first adding an idle behaviour, followed by a scouting behaviour and a foraging behavour.

Each of these behaviours are for the drones to collect resources for the mothership.

Following those behaviours, an interception stage was added, where the player would be attached when too close to the mothership.

We were tasked with adding our own additions, in which I just adjusted the laser spawn position, added fuel and a healing mechanic, where fuel and health values increased when idle.

Oh and I also added a maximum load or capacity for what the drones collected at the asteroids.

Possible Improvements

If I were to work on this project now, I would likely scrap the base project entirely as my biggest issues was with the assets used.

The base project felt over detailed and clunky, whereas a minimal pacman clone and a minimal wasp or bee fighting game for the second would demonstrate the same algorithms while being more pleasurable to actually play.

CAB401

Overview

A project of our choice was to be analysed for its performance and parallelized where possible; I chose Sebastian Lague's Marching Cubes project.

He has an entertaining video explaining the process he undertook and his view on the project.

What was learnt

Several perfomance analysis techniques were learnt. This was also my introduction into profiling.

A large problem I had was that the project was already extremely well performed, the proper use of compute shaders was implemented and it already had parallel processes implemented.

Chunk generation for performance increases along with the proper implementation of this system was also learnt.

Development of the project

A large portion of this assessment item is in the report itself, where a breakdown of the original project was made followed by a performance increase.

I had not done perfomance increases for this project but I did analyse the original and document it's algorithms, components and details.

I removed the chunk generation to push performance and see what chunk generation distance could theoretically be achieved.

A UML diagram was established detailing the classes used and a detailed technical report was written showing time efficiency graphs and profiles for various chunk generations.

Possible Improvements

I believe this project could genuinely have been improved, but perhaps not through further parallelization.

One example of this is rewriting the project to be of a smaller scope and removing existing features that aren't necessary (such as the terrain gradient).

Overall, I should have done a short analysis on the existing project beforehand to ensure that there is reasonable room for perfomance improvements.

I did this project quite early on in my degree, so I wasn't particularly familar with the idea of a looping project - moreso building one that remains functional.

CAB301

Overview

CAB301 was (in my opinion) a more advanced version of CAB201, in that a sorting algorithm was to be implemented along with a binary tree.

What was learnt

Binary trees and the difference between a few sorting algorithms (bubble sort, merge sort, etc)

Development of the project

The main component of this project was the sorting algorithm used. I had chose bubble sort because it's fun, simple and effective. For such a use case like the one provided, a fast algorithm wasn't needed

We were given interfaces, of which were not straight forward and we were told later into the semester to ignore them.

Outside of that, the only real error I had made was not properly using the Tool Library System as intended and not splitting my code between classes properly.

The main differences from CAB201 was that I did not do an input loop, but had it integrated in the main loop, where each option would run a different method.

This project also gave me a sneak peak into what CAB401 would be about, where an empirical analysis was done showing the O notation efficiency of the sorting algorithm used. (O(n^2))

Oh I should mention the sorting algorithm was just to sort tables alphabetically or by value before printing (depending on user preference)

Possible Improvements

Similarly to CAB201, a more colourful CLI would have helped.

If it were allowed I would have also used a library for prettier table printing since these tables were occasionally larger than the window.
